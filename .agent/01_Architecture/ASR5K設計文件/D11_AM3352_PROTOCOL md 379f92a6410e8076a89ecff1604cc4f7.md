# D11_AM3352_PROTOCOL.md

## 1. Purpose

[ARCHITECTURE]

本文件定義 ASR5K AM3352 與 C2000 F28384D 之間的 SPIB 通訊協定評估架構。

本文件目的：

- 定義 AM3352 → C2000 的控制命令格式
- 定義 C2000 → AM3352 的狀態回傳基礎規則
- 保留 Legacy Register-Based Protocol
- 新增 Packet Protocol Candidate
- 確保兩種 Protocol 功能一致
- 支援開發板模擬 AM3352 進行實測比較
- 作為 D12_SYSTEM_STATE_MACHINE.md 的上游輸入

## 4. Protocol Evaluation Strategy

[ARCHITECTURE]

D11 v0.1 定義兩組互斥的候選協定。

```
Candidate A:
Legacy Register-Based Protocol

Candidate B:
Packet Protocol
```

兩者皆需支援相同功能。

```
Output Control
DDS Control
Wave Page Select
Wave Download
Wave Validation
Wave Activation
Fault Clear
Status Readback
```

[ARCHITECTURE]

最終產品只選擇一組 Protocol。

```
Runtime protocol mixing is not required.
```

開發階段可透過開發板模擬 AM3352，分別測試 Candidate A 與 Candidate B。

## 5. SPIB Hardware Role

[DOCUMENTED]

SPIB 作為 AM3352 對 C2000 的主控通訊介面。

```
AM3352 = SPI Master
C2000  = SPI Slave
```

資料方向：

```
AM3352 MOSI
↓
SPIB RX FIFO
↓
DMA CH3
↓
RX Buffer
↓
Parser
↓
Internal Command Event
```

回傳方向：

```
System Status / Response
↓
TX Ping/Pong Buffer
↓
DMA CH4
↓
SPIB TX FIFO
↓
AM3352 MISO
```

## 6. Candidate A: Legacy Register-Based Protocol

### 6.1 Frame Format

[DOCUMENTED]

Legacy Register-Based Protocol 採用固定 32-bit frame。

```
Word0 = Address / Command
Word1 = Data16
```

或表示為：

```
[31:16] Address / Command
[15:0 ] Data
```

範例：

```
0x0958 0x0003
```

代表：

```
WAVE_PAGE_SELECT
Page ID = 3
```

---

### 6.2 Register-Based Protocol Concept

[ARCHITECTURE]

在 Legacy Register-Based Protocol 中：

```
Address = Command ID
Data16  = Command Parameter
```

例如：

```
0x0900 0x0001
```

代表：

```
OUTPUT_ON request
```

---

### 6.3 Legacy Wave Download

[DOCUMENTED]

Wave sample download 使用 address window：

```
0x3000 ~ 0x3FFF
```

每一筆 frame 對應一個 sample：

```
Address = 0x3000 + sample_index
Data16  = sample_data
```

範例：

```
0x3000 0x1234
0x3001 0x5678
0x3002 0xABCD
```

代表：

```
WaveSample[0] = 0x1234
WaveSample[1] = 0x5678
WaveSample[2] = 0xABCD
```

---

## 7. Candidate B: Packet Protocol

### 7.1 Packet Format

[ARCHITECTURE]

Packet Protocol Candidate 採用：

```
Header
Command ID
Payload Length
Payload
Checksum
```

目前暫定 word-based format：

```
Word0        Header
Word1        Command ID
Word2        Payload Word Count
Word3~WordN  Payload
Last Word    Legacy Checksum
```

---

### 7.2 Header

[ARCHITECTURE]

暫定：

```
Header = 0x55AA
```

用途：

```
Packet Start Detection
Frame Resynchronization
Protocol Identification
```

---

### 7.3 Command ID

[ARCHITECTURE]

Packet Protocol 的 Command ID 沿用 Register Address。

例如：

```
0x0900 = OUTPUT_ON
0x0958 = WAVE_PAGE_SELECT
0x3000 = WAVE_WRITE_BLOCK
```

目的：

```
Keep command semantic aligned with Legacy Register-Based Protocol.
```

---

### 7.4 Payload Length

[ARCHITECTURE]

Payload Length 暫定使用：

```
Payload Word Count
```

原因：

```
SPIB transfer unit = 16-bit word
C2000 parser unit = uint16_t
DMA buffer unit = uint16_t
```

範例：

```
Length = 0x0001
```

表示：

```
Payload contains 1 word
```

---

### 7.5 Payload

[ARCHITECTURE]

Payload 為：

```
uint16_t payload[]
```

不同 Command ID 對應不同 payload layout。

---

### 7.6 Checksum

[ARCHITECTURE]

Packet Protocol v0.1 暫定採用：

```
Legacy Checksum
```

Checksum 實際公式可於實作測試階段調整。

[ASSUMPTION]

暫定 checksum 涵蓋：

```
Header
Command ID
Payload Length
Payload Words
```

## 8. Common Command Address Allocation

[ARCHITECTURE]

D11 v0.1 暫定以下 command allocation。

| Address | Command | Data / Payload | Note |
| --- | --- | --- | --- |
| 0x0900 | OUTPUT_ON | 0x0001 | Request output enable |
| 0x0901 | OUTPUT_OFF | 0x0001 | Request output disable |
| 0x0910 | DDS_FREQ | uint16_t | Frequency command |
| 0x0911 | DDS_AMP | uint16_t PU | Amplitude command |
| 0x0912 | DDS_OFFSET | uint16_t PU | Offset command |
| 0x0958 | WAVE_PAGE_SELECT | page_id | Select target wave page |
| 0x0960 | WAVE_VALIDATE | page_id or selected page | Validate wave page |
| 0x0961 | WAVE_ACTIVATE | page_id or selected page | Activate valid wave page |
| 0x0970 | CLEAR_FAULT | 0x0001 | Clear test fault |

## 9. DDS Command Semantics

### 9.1 DDS_FREQ

[ARCHITECTURE]

```
Command = 0x0910
```

Register-Based:

```
Word0 = 0x0910
Word1 = frequency_value
```

Packet-Based:

```
Header  = 0x55AA
Command = 0x0910
Length  = 1
Payload = frequency_value
Checksum
```

[ASSUMPTION]

D11 v0.1 不強制最終 frequency unit。

候選：

```
Option A:
frequency_hz

Option B:
frequency_x100

Option C:
frequency_pu
```

正式單位由 protocol test report 與 D12/DDS integration 決定。

---

### 9.2 DDS_AMP

[ARCHITECTURE]

```
Command = 0x0911
```

Payload：

```
uint16_t amplitude_pu
```

暫定語意：

```
0x0000 = 0.0 PU
0xFFFF = 1.0 PU
```

---

### 9.3 DDS_OFFSET

[ARCHITECTURE]

```
Command = 0x0912
```

Payload：

```
uint16_t offset_pu
```

暫定語意：

```
0x8000 = center offset
```

## 10. Wave Management Commands

### 10.1 WAVE_PAGE_SELECT

[DOCUMENTED]

```
Command = 0x0958
Data    = Page ID
```

Register-Based example:

```
0x0958 0x0003
```

Packet-Based example:

```
0x55AA
0x0958
0x0001
0x0003
checksum
```

Internal event:

```
CMD_WAVE_SELECT_PAGE(page_id = 3)
```

---

### 10.2 WAVE_WRITE_SAMPLE

[DOCUMENTED]

Legacy Register-Based wave sample write uses:

```
0x3000 ~ 0x3FFF
```

Mapping:

```
sample_index = address - 0x3000
```

---

### 10.3 WAVE_WRITE_BLOCK

[ARCHITECTURE]

Packet Protocol supports multi-sample wave download.

Packet format:

```
Header
Command ID = 0x3000
Length = N + 1
Payload[0] = start_index
Payload[1] = sample[start_index]
Payload[2] = sample[start_index + 1]
...
Payload[N] = sample[start_index + N - 1]
Checksum
```

Example:

```
0x55AA
0x3000
0x0041
0x0100
sample[0x0100]
sample[0x0101]
...
sample[0x013F]
checksum
```

Meaning:

```
Write 64 samples starting from sample_index 0x0100.
```

---

### 10.4 WAVE_VALIDATE

[DOCUMENTED]

```
Command = 0x0960
```

Purpose:

```
Request validation of selected wave page.
```

Validation shall check:

```
Page ID valid
Address coverage
Sample count
Checksum
Download complete condition
Output state
```

---

### 10.5 WAVE_ACTIVATE

[DOCUMENTED]

```
Command = 0x0961
```

Activation condition:

```
Wave Page Valid
Output OFF
```

If activation fails, the current active wave page shall remain unchanged.

## 11. Output Protection Rule

[DOCUMENTED]

Output OFF allows:

```
Wave Download
Wave Validation
Wave Activation
Flash Save
Flash Load
```

Output ON allows:

```
Status Read
Fault Read
Wave Readback
```

Output ON forbids:

```
Wave Download
Wave Modification
Wave Activation
Flash Write
Flash Erase
OTA Update
```

D11 command parser shall reject forbidden commands according to current system state.

Validation result

## 14. Internal Command Event Layer

[ARCHITECTURE]

Both protocol candidates must map into the same Internal Command Event Layer.

```
Legacy Register Parser
↓
Internal Command Event
↓
D12 System State Machine
```

```
Packet Parser
↓
Internal Command Event
↓
D12 System State Machine
```

Example:

```
Legacy:
0x0958 0x0003

Packet:
0x55AA 0x0958 0x0001 0x0003 checksum
```

Both map to:

```
CMD_WAVE_SELECT_PAGE(page_id = 3)
```

This rule prevents D12 and application logic from depending on the external protocol format.

## 15. Protocol Parser Rules

### 15.1 Legacy Parser

[ARCHITECTURE]

Legacy parser receives fixed 2-word frame.

```
Address
Data16
```

Parser action:

```
Decode address
Validate data
Generate internal command event
```

---

### 15.2 Packet Parser

[ARCHITECTURE]

Packet parser receives variable-length packet.

```
Header
Command ID
Length
Payload
Checksum
```

Parser action:

```
Search header
Read command ID
Read payload length
Receive payload
Verify checksum
Generate internal command event
```

---

### 15.3 Invalid Packet Behavior

[ASSUMPTION]

暫定規則：

```
Invalid header:
Drop current word and search next header.

Invalid length:
Drop packet.

Checksum error:
Drop packet.
Set test fault code = CHECKSUM_ERROR.
Do not generate internal command event.
```

此規則可於 protocol test report 後調整。

---

## 16. DMA Considerations

[DOCUMENTED]

SPIB RX uses DMA CH3.

SPIB TX uses DMA CH4.

Runtime DMA ownership:

```
CH1 = SPIC TX
CH2 = SPIC RX
CH3 = SPIB RX
CH4 = SPIB TX
CH5 = SPIA RX
CH6 = SPIA TX
```

Runtime rule:

```
CH1 ~ CH4 reserved for runtime.
CH5 ~ CH6 reserved for boot / OTA / maintenance.
```

[ARCHITECTURE]

D11 v0.1 requires both protocol candidates to be tested using AM3352 simulator board.

| Test Item | Legacy Register-Based | Packet Protocol | Purpose |
| --- | --- | --- | --- |
| Basic command | 0x0900 OUTPUT_ON | Header + 0x0900 + Payload + Checksum | Verify command equivalence |
| Output OFF | 0x0901 OUTPUT_OFF | Header + 0x0901 + Payload + Checksum | Verify output control |
| DDS frequency | 0x0910 DDS_FREQ | Header + 0x0910 + Payload + Checksum | Compare DDS command handling |
| DDS amplitude | 0x0911 DDS_AMP | Header + 0x0911 + Payload + Checksum | Compare PU parameter handling |
| DDS offset | 0x0912 DDS_OFFSET | Header + 0x0912 + Payload + Checksum | Compare offset command |
| Wave page select | 0x0958 + page_id | Header + 0x0958 + page_id + Checksum | Verify page selection |
| Wave download | 0x3000~0x3FFF one sample/frame | 0x3000 block packet | Compare throughput |
| Wave validation | 0x0960 | Header + 0x0960 + Payload + Checksum | Verify validation flow |
| Wave activation | 0x0961 | Header + 0x0961 + Payload + Checksum | Verify activation flow |
| Checksum error | Legacy checksum behavior | Packet checksum behavior | Compare error detection |
| Lost frame recovery | Address continuity | Header + Length + Checksum | Compare recovery |

## 18. Test Success Criteria

[ARCHITECTURE]

Both protocol candidates shall pass:

```
1. OUTPUT_ON command accepted only when system state allows.
2. OUTPUT_OFF command works from OUTPUT_ON state.
3. DDS_FREQ / DDS_AMP / DDS_OFFSET command updates DDS target.
4. WAVE_PAGE_SELECT selects target wave page.
5. 4096 samples can be downloaded.
6. Download Complete / Validation / Activation flow works.
7. Invalid wave data cannot be activated.
8. Output ON rejects wave modification.
9. Status / fault readback works.
10. No RX FIFO overrun during wave download test.
```
