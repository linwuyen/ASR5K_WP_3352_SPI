# [音檔說明]關於 SPI RX（接收端）封包格式與 DMA 機制技術說明

這段討論是基於一個 SPI 封包從 **Master/Slave (例如從某個 SPI 晶片傳送到 3352 處理器)** 的傳輸規劃。為了在多晶片的 Bus（匯流排）線上準確接收大筆資料，設計了一套完整的封包結構。

### 1. SPI RX 封包格式定義 (Packet Format)

一個完整的基本通訊封包，由以下幾個部分組成：

| 欄位名稱 | 功能說明 | 資料長度 (以 Word 為單位) |
| --- | --- | --- |
| **Header (標頭)** | 封包的起頭標誌，讓接收端識別封包從哪裡開始。 | 1 Word |
| **Command ID** | 命令識別碼。當 Bus 線上掛載多個晶片時，用來識別與執行特定的功能。系統預留了 65,000 多個命令空間供設計使用。 | 1 Word |
| **Length (資料長度)** | 用來指定後面接著的 **Payload** 實際有多少個 Word。 | 1 Word |
| **Payload (資料內容)** | 實際要傳輸的數據。長度由 Length 決定（例如長度設為 16，後面就有 16 組 Word）。 | 可變長度 (N Words) |
| **CRC (校驗碼)** | 用於檢查整段封包在傳輸過程中是否發生錯誤。 | 1 Word |

> **基本封包長度計算：**
如果 Payload 只有 **1 個 Word**，那麼整包封包總共包含：Header(1) + Command ID(1) + Length(1) + Payload(1) + CRC(1) = **5 个 Words**。
在 16-bit 系統下，這相當於 5 \times 16 = 80 \text{ bits}；而在音訊中提及的基本單位下，若算上基本控制開銷，最少也需要傳輸數十個位元（如 30~32 bits 以上，依硬體定寬而定）。當 Payload 增加到 16 個 Word 時，封包大小就會大幅超過硬體基本 FIFO 的限制。
> 

### 2. 為什麼 RX 接收端必須採用硬體 DMA？

當 SPI 開始連續灌入大筆資料（Payload 變長）時，硬體 FIFO 很快就會被塞滿。此時，**「底層硬體 DMA 的機制必須寫好」**：

- **即時抽走資料**：每當硬體 FIFO 一接收到資料，**DMA 就必須在底層自動、無條件地把資料從 FIFO 搬移到記憶體緩衝區（RAM）中抽走**，避免 FIFO 滿載導致後續資料遺失（Overrun Error）。
- **與 TX Ping-Pong 機制遙相呼應**：正因為 RX 端有這種大筆資料連續灌入的特性，如果未來 TX 端（傳送端）也要用同樣長的封包格式將資料回傳（例如回傳整組校正數據），TX 端就必須像前一節提到的，採用 **Ping-Pong Buffer** 的硬體交替機制，才能確保傳送時不會受限於 FIFO 的容量限制。

### 3. SPI 接收通訊協定架構 (Doxygen 規範)

為了將此封包格式規範化，以下提供符合業界標準的 Doxygen 結構體定義，將 Header、Command、Length、Payload 與 CRC 完整封裝：
c
/**

- @file spi_rx_protocol.h
- @brief SPI RX 接收端封包格式與通訊協定定義
- @note 支援可變長度 Payload 與底層 DMA 自動接收
*/
#ifndef SPI_RX_PROTOCOL_H
#define SPI_RX_PROTOCOL_H
#include <stdint.h>
/*
- @brief SPI 接收封包解析狀態機 (FSM)
- @details 用於解析 RX 接收到的連續資料流
*/
typedef enum {
RX_STAGE_SEARCH_HEADER, /**< 00: 尋找封包起頭 Header */
RX_STAGE_GET_COMMAND, /**< 01: 讀取 Command ID */
RX_STAGE_GET_LENGTH, /**< 02: 讀取 Payload 長度 */
RX_STAGE_GET_PAYLOAD, /**< 03: 依據 Length 讀取數據內容 */
RX_STAGE_CHECK_CRC /**< 04: 驗證 CRC 校驗碼 /
} RX_FSM_STAGE;
/*
- @brief SPI 標準通訊封包結構體
- @note 命名規範：型態 + 長度 + 名稱
*/
typedef struct {
uint16_t u16_Header; /**< 16-bit 封包起頭標誌 (Packet Start) */
uint16_t u16_CommandID; /**< 16-bit 功能命令識別碼 (支援多晶片定址) */
uint16_t u16_Length; /**< 16-bit 後續 Payload 的 Word 數量 */
uint16_t *pu16_Payload; /**< 指向實際數據緩衝區的指標 (可變長度) */
uint16_t u16_CRC; /**< 16-bit 錯誤檢查校驗碼 */
} STRUCT_SPI_PACKET;
#endif /* SPI_RX_PROTOCOL_H */

### 4. 總結

本段討論的核心在於**「建構標準化的通訊封包格式」**。透過明確的 **Header、Command ID、Length、Payload、CRC** 劃分，讓系統不論是面對單筆資料還是 16 筆以上的 Block 資料，都能透過底層 **DMA** 穩定接收；同時也確認了未來 TX 端為了匹配這種大封包，導入 **Ping-Pong Buffer** 的必要性。
