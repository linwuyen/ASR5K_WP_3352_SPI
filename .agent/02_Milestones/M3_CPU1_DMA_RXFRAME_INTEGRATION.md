# M3_CPU1_DMA_RXFRAME_INTEGRATION Architecture Contract (PASS / CLOSED)

---

## 1. Milestone Scope

M3 integrates CPU1 SPIB RX DMA reception only.
The target data path is:
```text
SPIB RX FIFO
     ↓
  DMA CH3
     ↓
2-word RxFrame Ping/Pong (Ping = 2 words, Pong = 2 words)
     ↓
CPU1 Background Parser
```

---

## 2. Architecture Rules

*   **Register-Frame Compatibility**: M3 shall preserve the existing SPIB register-frame protocol (`RxFrame_t`: `cmd` + `data` 16-bit registers).
*   **No Protocol Changes**: M3 shall not introduce any packet-style wrapping, header magic, or framing protocol.
*   **System Boundaries**: M3 shall not modify:
    *   CPU2
    *   IPC
    *   MSGRAM
    *   EMIF1
    *   DDS
    *   Flash

---

## 3. DMA Ownership

*   **DMA CH3 Exclusive Owner**: DMA Channel 3 is exclusively owned by SPIB RX and shall not be reused for other runtime functions.
*   **Trigger Source**: `DMA_TRIGGER_SPIBRX` is generated by the SPIB RX FIFO event. FIFO trigger level shall be configured so that one complete 2-word register frame can be transferred by DMA CH3.
*   **Data Size**: 16-bit (`DMA_CFG_SIZE_16BIT`).
*   **Burst Settings**:
    *   Burst Size: 2 words (1 frame: `cmd` + `data`).
    *   Source Burst Step: `0` (Source is the fixed register `SPIB_SYSTEM_BASE + SPI_O_RXBUF`).
    *   Destination Burst Step: `1` (Destination increments by 1 word for each word in the burst).
*   **Transfer Settings**:
    *   Transfer Size: `1` (1 burst per transfer = 2 words total).
    *   Source Transfer Step: `0`.
    *   Destination Transfer Step: `0`.
*   **DMA Completion Detection**:
    *   Polled in the background loop by checking `DMA_getRunStatusFlag(DMA_CH3_BASE) == false`.
    *   **No DMA CH3 ISR shall be enabled or implemented for M3.**

---

## 4. Buffer Ownership

RxFrame Ping/Pong ownership is split as follows:
*   **DMA-owned buffer**: The 2-word buffer currently receiving the SPIB frame.
*   **CPU-owned buffer**: The 2-word buffer containing a completed frame waiting for/undergoing background parsing.

*   **Access Rules (Mutual Exclusion)**:
    *   DMA must never write into the buffer currently being parsed by CPU1.
    *   CPU1 must never parse the buffer currently owned by DMA.
    *   The buffer size for Ping and Pong is exactly 2 words (1 legacy register frame).

---

## 5. Parser Rule

*   **Background Context**: The CPU1 parser runs only in the background context (called cooperatively from the main loop via `runSPIBslave()`).
*   **Trigger**: Parser consumes completed 2-word RxFrame buffers after DMA completion is detected.
*   **No ISR Parsing**: No SPIB RX byte-level or word-level ISR shall be used for parsing.

---

## 6. Timing Rule

*   **Jitter Avoidance**: SPIB RX reception must not introduce jitter into the 100 kHz real-time runtime path.
*   **Hardware Offloading**: DMA CH3 performs all hardware data movement autonomously.
*   **Non-Critical Scheduling**: CPU1 background parser performs non-critical interpretation in the cooperative main loop.

---

## 7. Completion Rule

*   **Completion Criteria**: A frame is considered ready only when the DMA CH3 transfer-complete status is observed via polling in the background loop.
*   **Alternate Arming**: After CPU1 accepts the completed buffer, DMA CH3 is armed to the alternate buffer in the background loop.

---

## 8. Error Policy

*   **Overrun / Drop Condition**: If both Ping and Pong are unavailable, M3 shall report an RxFrame overrun/drop condition.
*   **Buffer Pressure Fault**: If DMA completion occurs while the parser still owns the previous completed buffer, M3 shall mark a buffer pressure fault.
*   **Isolation Policy**: M3 shall not attempt recovery by touching unrelated subsystems (e.g. resetting CPU2, resetting EMIF1, or altering IPC).

---

## 9. Acceptance Criteria

M3 passes when:
1.  SPIB RX FIFO data is moved by DMA CH3.
2.  RxFrame Ping/Pong alternates correctly between 2-word Ping and Pong buffers.
3.  CPU1 background parser receives complete 2-word frames.
4.  No CPU2 / IPC / MSGRAM / EMIF1 / DDS / Flash code is modified.
5.  No packet protocol is introduced.
6.  No SPIB RX byte-level ISR parsing is introduced.

---

## 10. Non-Goals

M3 does not implement:
*   Wave download to EMIF1
*   DDS parameter update
*   Flash save/load
*   IPC notification
*   CPU2 command forwarding
*   SPIB TX response redesign
*   Protocol migration
*   DMA CH3 ISR
*   Header / Length / Payload / CRC packet parser
*   64-word RX buffer
*   Variable-length packet
*   Recovery engine

## Appendix: Legacy Register Frame vs Packet Protocol Comparison

This appendix is for documentation purposes only. The Packet Protocol is marked strictly as a **Future Candidate** and is not part of the active M3 design.

| Comparison Item | Legacy Register Frame (Active M3 Design) | Packet Protocol (Future Candidate) |
|---|---|---|
| **Frame size** | Fixed 2 words (16-bit CMD + 16-bit DATA) | Variable-length packet structure |
| **Header / Framing** | None | Header magic, length field, payload, CRC |
| **DMA Buffer Size** | 2-word Ping/Pong buffers | 64-word (or larger) RX buffer |
| **Parsing overhead** | In-place direct mapping (very low) | Header validation, CRC computation |
| **Implementation Complexity** | Minimal | Moderate to High |
| **Use case** | Low-latency register-based command transfer | Block data transfer / Waveform table downloads |

### Conclusion

M3 active design remains Legacy Register Frame.
Milestone M3 is officially **PASS / CLOSED**.
