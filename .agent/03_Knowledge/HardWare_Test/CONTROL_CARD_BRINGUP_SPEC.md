# Control Card Hardware Bring-up Specification (SDD)

本文件定義新製作 PCB Control Card (TI C2000 系列) 的標準檢查與驗證流程。所有新硬體必須通過此程序方可進入應用層韌體開發。

## 1. 範疇 (Scope)
基於 TMS320F28377S/D 或 TMS320F28388D 的 Control Card。

## 2. 核心通關清單 (Critical Path - MVP)
若時間緊迫，請至少完成以下五項「Go/No-Go」檢查：

| 序號 | 檢查重點 | 預期結果 | 實測值 (Measured) | 失敗風險 |
| :--- | :--- | :--- | :--- | :--- |
| **C1** | **電源對地阻抗** | 3.3V > 100Ω, 1.2V > 10Ω | 3.3V: 1k, 1.2V: 48 | **燒毀 DSP / LDO** |
| **C2** | **LDO 輸出電位** | 3.3V / 1.2V 誤差 < ±3% | 3.3V / 1.2V (準確) | 核心不工作或壽命縮短 |
| **C3** | **基準電壓 (REF/MID)** | 3.000V / 1.500V 誤差 < ±5mV | VREFAB: 5.2k (Cold) | **控制環路失控/偏移** |
| **C4** | **JTAG 連線** | CCS "Test Connection" Passed | Success (Flashable) | DSP 損壞或 Reset 被鎖定 |
| **C5** | **主動時鐘 (CLK)** | 25.0 MHz 穩定方波 | Success (Blinky OK) | 系統無法啟動 |

## 11. Agent 執行檢查規範 (Agent Verification Protocol)
當 Agent 協助開發時，必須遵循以下「硬體意識」檢查：

1. **查閱 SysConfig**: 在修改任何外設代碼前，Agent 必須先讀取 `pinmux.syscfg`，確認腳位功能。
2. **暫存器比對**: Agent 應主動比對代碼中的暫存器配置與 TRM (Technical Reference Manual) 的描述。
3. **安全屏障檢查**: 若代碼涉及 PWM 或 ADC，Agent 必須檢查是否有 `#pragma CODE_SECTION` 將 ISR 放於 RAM 中，以及死區時間是否符合 `HwConfig.h`。

## 12. 軟硬體邊界與三層架構對齊 (SW/HW Boundary & Three-Layer Alignment)
硬體驗證必須嚴格遵循專案之三層解耦架構 (Three-Layer Architecture)：
- **HAL 層驗證**: 所有上述涉及 GPIO、PWM、ADC 的驗證碼，必須限制在 DriverLib / 暫存器配置範疇，強制使用 `EALLOW` / `EDIS` 包覆。嚴禁將驗證用的硬體相關配置滲透至 Control 層。
- **Module 層資料傳遞**: 用於量測回授的驗證碼，需透過純數值結構體指標傳遞 ADC 數值。
- **類型與命名**: 所有撰寫的驗證用 `.c` / `.h`，必須 100% 遵守 `STRICT_STDINT` (使用 `uint16_t`, `float32_t`，禁用 `int`) 以及 `ASCII_ONLY_CODE` (代碼與註釋全英文)。

## 13. 硬體教訓與開發心得 (Lessons Learned)
在 ASR5K F28384D Bring-up 過程中發現的關鍵物理限制：

1. **SPI Flash 必須使用手動 CS**: C2000 硬體 SPISTE 腳位在 Master 模式下會在每 8-bit 間隙拉高，這會導致 SPI Flash (W25Q系列) 通訊重置。必須將 CS 改為一般 GPIO 並手動拉低。
2. **多工覆寫與 EALLOW**: 若 SysConfig 鎖定了某個腳位的功能，在 `main.c` 中手動覆寫 Mux 時必須包裹在 `EALLOW` 與 `EDIS` 之間，否則設定無效。
3. **SPIA vs SPIB 誤判**: 確認外接 Flash 物理連線為 **SPIA (GPIO 16, 17, 18, 19)**。之前的 SPIB (GPIO 60, 61, 65, 66) 為其他用途，誤用會導致讀值全為 1。
